### 切换运行级别

```Shell
init 0          # 关机
init 1          # 单用户模式（维护）：只有 root 用户
init 2          # 默认：完整的系统服务（网络、数据库等）
init 3          # 切换到文本模式
init 5          # 切换到图形模式
init 6          # 重启
telinit 3       # 同 init（某些系统）
```

### 服务管理

```Shell
systemd
systemctl

# 查看状态
systemctl status nginx           # 查看服务状态（是否运行、进程PID、最近日志）
systemctl is-active nginx        # 只看是否运行（返回 active/inactive，脚本里用）
systemctl is-enabled nginx       # 只看是否开机自启（返回 enabled/disabled）
systemctl list-units --type=service  # 列出所有正在运行的服务
systemctl list-units --all --type=service  # 列出所有服务（含停止的）

# 起停管理
systemctl start nginx            # 立即启动
systemctl stop nginx             # 立即停止
systemctl restart nginx          # 重启（不管服务在不在都先停再起）
systemctl reload nginx           # 平滑重载配置（不中断服务，仅部分服务支持）
systemctl reload-or-restart nginx  # 能reload就reload，不能就restart

# 开机自启
systemctl enable nginx           # 设置开机自启（创建软链接到 /etc/systemd/system/multi-user.target.wants/）
systemctl disable nginx          # 取消开机自启
systemctl enable --now nginx     # 设置自启+立即启动（合体技，省一行命令）

3 屏蔽/解屏蔽（慎用）
systemctl mask nginx             # 彻底锁死服务（任何人都不能启动，软链接到 /dev/null）
systemctl unmask nginx           # 解锁



【三、查看日志（journalctl）—— systemd 自带的日志系统】

journalctl -u nginx              # 查看 nginx 服务的所有日志
journalctl -u nginx -f           # 实时跟踪日志（类似 tail -f）
journalctl -u nginx -n 100       # 查看最近 100 行
journalctl -u nginx --since "2026-06-20 10:00:00"  # 查指定时间之后的日志
journalctl -u nginx --since "1 hour ago"           # 查最近1小时的日志
journalctl -u nginx -p err       # 只看错误级别以上的日志

【排错必杀技】服务起不来，第一件事：
systemctl status nginx           # 看状态里的报错信息
journalctl -u nginx -n 50        # 看最后50行日志，基本能定位 90% 问题


【四、服务配置文件（单元文件 .service）】

【位置】（优先级从高到低）
/etc/systemd/system/nginx.service     # 用户自定义/修改的（优先级最高）
/run/systemd/system/nginx.service     # 运行时临时文件（重启丢）
/lib/systemd/system/nginx.service     # 系统默认安装的（软件包自带）

【运维铁律】要改配置，把 /lib/systemd/system/xxx.service 复制到 /etc/systemd/system/ 再改，这样软件包升级时不会被覆盖。

【配置文件核心字段（看一眼就懂）】
[Unit]
Description=Nginx Web Server          # 服务描述
After=network.target                  # 在 network 启动之后才启动
Wants=network.target                  # 尽量依赖 network

[Service]
Type=forking                          # 启动方式（forking=后台守护进程，simple=前台进程）
PIDFile=/run/nginx.pid                # PID文件位置（Type=forking 必须有）
ExecStart=/usr/sbin/nginx -c /etc/nginx/nginx.conf   # 启动命令
ExecReload=/usr/sbin/nginx -s reload                 # 重载命令（不写则 reload 无效）
ExecStop=/usr/sbin/nginx -s quit                     # 停止命令
Restart=on-failure                    # 挂掉后自动重启（=always 一直重启，=on-failure 仅异常退出时重启）
RestartSec=5s                         # 重启前等待5秒

[Install]
WantedBy=multi-user.target            # 在 multi-user 运行级别下启用（即开机启动）

【改完配置必须执行】：
systemctl daemon-reload               # 重新加载所有服务配置文件（改了 .service 必执行！）


【五、运行级别（target）—— 系统启动的不同“模式”】

【systemd target 对照老 SysV runlevel】
runlevel 0 = poweroff.target      # 关机
runlevel 1 = rescue.target        # 单用户救援模式
runlevel 3 = multi-user.target    # 多用户命令行模式（服务器默认）
runlevel 5 = graphical.target     # 图形界面模式
runlevel 6 = reboot.target        # 重启

【切换运行级别】
systemctl get-default             # 查看当前默认启动模式
systemctl set-default multi-user.target  # 设置为命令行模式启动（服务器标配）
systemctl isolate multi-user.target      # 立即切换到命令行模式（不用重启）

【查看当前系统启动耗时】
systemd-analyze                    # 看整体启动时间
systemd-analyze blame              # 看每个服务启动耗时（排慢启动的神器！）
systemd-analyze critical-chain     # 看关键链，找出拖慢启动的元凶


【六、运维必踩的 5 个大坑】

坑1：改了配置文件忘了执行 systemctl daemon-reload
# 现象：改了 /etc/systemd/system/nginx.service，然后 restart，发现没生效
# 原因：systemd 缓存没更新
# 解决：修改任何 .service 文件后，必须执行 systemctl daemon-reload

坑2：reload 以为能生效，但服务根本不支持 reload
# 现象：执行 systemctl reload nginx，返回失败
# 原因：有些服务（如 MySQL）不支持平滑重载
# 解决：用 systemctl restart nginx

坑3：systemctl mask 和 systemctl disable 搞混
# disable = 取消开机自启，但手动 start 还能起
# mask = 彻底锁死，手动 start 也起不来（软链接到 /dev/null）
# 线上别轻易 mask，尤其是系统关键服务（如 sshd）

坑4：Restart=always 导致服务无限重启
# 如果程序启动失败（如配置文件语法错误），systemd 会不断尝试重启，日志刷屏
# 解决：调试时先把 Restart=no 改掉，或者加 RestartSec=30 间隔长一点

坑5：用 systemctl stop 杀不死进程（僵尸/孤儿进程）
# 现象：systemctl stop nginx 执行后，ps aux | grep nginx 还有残留
# 原因：程序 fork 出子进程没被正确回收（Type=forking 没配好 PIDFile）
# 解决：
1. 检查 .service 文件里 Type 和 PIDFile 是否配对了
2. 紧急情况下用 pkill -9 nginx，然后 systemctl start nginx 重新拉起来


【七、调试三板斧（服务起不来照着查）】

1. systemctl status 服务名
   # 看状态里的报错信息，Active: failed 后面通常跟着具体原因

2. journalctl -u 服务名 -n 50
   # 看服务自己的日志，程序崩溃通常有明确报错

3. 直接执行 ExecStart 里的命令（绕过 systemd）
   # /usr/sbin/nginx -c /etc/nginx/nginx.conf
   # 手动执行看终端输出什么错误（环境变量、路径问题一目了然）


【八、特殊技巧（生产实用）】

【开机自动启动某个脚本（自己写的简单服务）】
# 新建 /etc/systemd/system/myscript.service
[Unit]
Description=My custom script
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/my_script.sh
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target

# 然后
systemctl daemon-reload
systemctl enable myscript
systemctl start myscript

【临时增加系统资源限制（改 ulimit）】
# 在 .service 文件的 [Service] 段加：
LimitNOFILE=65535              # 最大文件句柄数（Nginx/数据库必备）
LimitNPROC=65535               # 最大进程数

【优雅停止 vs 强制杀死】
systemctl stop nginx           # 发送 SIGTERM，程序可自行清理（优雅）
systemctl kill nginx           # 发送 SIGKILL，暴力杀死（不到万不得已不用）
```
