> IP,网卡，网关路由端口，连通，DNS，路由，ssh，抓包，流量，网络基础与核心概念

```Shell
1. 用户反馈“访问不了”
   ↓
2. ping IP 通不通？
   ├── 不通 → 查路由（traceroute）→ 查网关/交换机/VPN
   └── 通 → 进入第3步
   ↓
3. telnet IP 端口 通不通？
   ├── 不通（超时）→ 查防火墙（firewalld/iptables/云安全组）
   ├── 不通（拒绝）→ 查服务（systemctl status）→ 查端口监听（ss -tulnp）
   └── 通 → 进入第4步
   ↓
4. 是域名访问吗？
   ├── 是 → 查 DNS（dig）→ 查 hosts 文件
   └── 否 → 查应用自身错误日志（/var/log/xxx/error.log）
   ↓
5. 如果能通但巨慢 → 查带宽（iftop）→ 查丢包（mtr）→ 查 MTU（ping -M）→ 查内核连接表
```

### 基础配置与测试

```Shell
# IP与网卡配置
ip -br a                          # 简洁查看所有网卡IP地址
ip a s [show eth0]                # 查看详细IP配置（网卡）
ethtool [-S] eth0                 # 查看网卡物理/驱动状态：Speed，Duplex（硬件统计）
ip -s link show eth0              # 查看网卡错误计数：RX errors / TX errors / dropped
ip link set eth0 up/down          # 配置网卡
ifup eth0  &&  ifdown eth0        # 手动启停网卡

# 网关路由配置
lspci | grep -i ethernet          # 查看网卡驱动和硬件信息
ip route show                     # 看本机路由表：default via 网关IP（排查网络不通）
ip neigh show                     # 查看ARP缓存表（IP与MAC的映射）
ip rule show                      # 查看策略路由规则
ip neigh del 192.168.1.1 dev eth0 # 手动删除某个错误的ARP记录（刷新后自动学习）

# 网络连通性测试
ping [-I eth0] -c 4 8.8.8.8       # 测试基础连通性（指定源IP）
# connect: Network is unreachable → 本机没有去往目标的路由（缺网关）
# 100% packet loss → 对方没回包（可能禁ping、路由不通、防火墙丢包）
# time=100ms，有丢包 → 线路质量差
ip route get 8.8.8.8              # 查看去往特定IP的路由走向（选路测试）
mtr -r -c 10 8.8.8.8              # 路由追踪（实时显示每一跳的延迟和丢包率）
```

### 传输与应用层

```shell
# 传输层与连接
ss -tulnp                         # 看所有监听的TCP/UDP端口及对应进程状态（确认服务起来了）
nc -zv 192.168.1.100 80           # 测远程端口通不通
# Connected to ... → 端口通的，服务在响应（问题在应用层）
# Connection refused → 端口没开，或服务没起，或防火墙返回了 RST 包
# Connection timed out → 路由不通，或防火墙直接丢弃了 SYN 包（没回应）
ss -antp                          # 查看已建立的TCP连接
lsof -i :80                       # 查看具体端口被哪个进程占用
iptables -L -n -v --line-numbers  # 查看防火墙iptables规则
ss -tlnp | grep 8080              # 查看具体端口占用
ss -m                             # 查看socket统计信息（内存占用等）

# 应用层与DNS
cat /etc/resolv.conf              # 查看本机DNS解析配置（指向哪个DNS服务器）
cat /etc/hosts                    # 检查 hosts 文件劫持
dig +short www.baidu.com          # 快速解析域名（+short只返回IP，干净利落）
dig @114.114.114.114 www.baidu.com
nslookup www.baidu.com            # 指定DNS服务器测（测试内网DNS是否正常）
time dig +short www.baidu.com     # 测试DNS解析耗时
curl -v http://192.168.1.100:80   # HTTP 服务测试
```

### 抓包分析

```Shell
tcpdump [-i eth0][-c 50][-nn][host 192.168.1.100]/[(tcp)port 80][-w /tmp/capture.pcap]  
#         网卡     数量  不解析        特定主机           特定端口        保存为pcap文件
# 如果看到只有 SYN（从客户端来），没有 SYN-ACK（从服务器回）→ 防火墙丢弃或服务未监听
# 如果看到 SYN -> SYN-ACK -> ACK（三次握手完整）→ 网络层没问题，查应用层
# 如果看到 RST 包 → 服务拒绝了连接（端口没开或者防火墙 reject）

# 抓取SYN包（排查握手失败，-v显示详细信息）
tcpdump -i eth0 -nn 'tcp[tcpflags] & (tcp-syn) != 0'
```

### TCP三次握手（建立连接）

```Shell
客户端 → 服务器：SYN（seq=x）
服务器 → 客户端：SYN+ACK（seq=y, ack=x+1）
客户端 → 服务器：ACK（seq=x+1, ack=y+1）
运维视角：(命令:tcpdump)
telnet IP port 如果能通             -> 说明三次握手成功（服务端端口已监听且无防火墙拦截）。
抓包看到只有SYN发出，没有SYN+ACK回来   -> 要么服务端没监听端口 要么防火墙丢弃了SYN包
抓包看到SYN之后收到RST               -> 端口被拒绝（服务没启动或主动拒绝）
抓包看到Retransmission              -> 网络丢包或延迟过大。
抓包看到Dup ACK（重复确认）           -> 对端收到乱序包，触发快速重传。

# 为什么要三次握手？
两次握手可能导致服务器因收到客户端滞后的旧SYN包而误建连接，浪费资源且状态不一致。
三次握手中，服务器必须等到客户端的ACK确认才会建立连接，从而避免这一问题。
```

### TCP四次挥手（断开连接）

```Shell
主动关闭方 → 被动方：FIN
被动方 → 主动方：ACK
被动方 → 主动方：FIN
主动方 → 被动方：ACK
(命令:ss)
运维常见状态   含义	                           常见原因
ESTABLISHED	 连接已建立	                     正常
TIME_WAIT	   主动关闭方在等待2MSL	             频繁短连接导致占用端口->调整内核参数tcp_tw_reuse
CLOSE_WAIT	 被动关闭方收到FIN，应用未调用close  应用bug，没及时关闭连接 →导致连接泄漏->修复应用代码
FIN_WAIT2	   主动关闭方等待对方FIN	             对端迟迟不关闭

# 为什么客户端需要等待TIME_WAIT？
客户端发送最后一个ACK后不立即释放，而是等待2MSL。
若ACK丢失，服务端会重发FIN，客户端收到后可重发ACK并刷新计时器，确保服务端正常关闭。
目的是在不可靠网络中可靠地确认连接断开，避免服务端一直停留在最后确认状态。
# 为什么要四次挥手？
TCP支持半关闭：每一方可以单独停止发送数据。
关闭一个方向需要“FIN + ACK”两次交互，双方共需四次挥手才能完全释放连接。
```
