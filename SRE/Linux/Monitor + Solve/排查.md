> 排障步骤

------

### 第一步

```Shell
第一阶段：现象确认与近期变更（1分钟）
-----------------------------
▶ 诊断动作：
   - 确认故障现象（500错误 / 连接超时 / 服务器无响应）。
   - 查看最近命令历史：history | tail -30
   - 查看最近修改的文件：find /etc /opt /app -mmin -10 -type f 2>/dev/null

✅ 排查到该问题时的处置措施：
   1. 若明确为代码变更导致 → 立即执行回滚（Git revert / 替换旧jar包 / 重启服务）。
   2. 若明确为配置变更导致 → 恢复备份配置文件（cp /etc/xxx.conf.bak /etc/xxx.conf）并重载服务。
   3. 若无法确认具体变更，进入下一阶段，优先重启业务进程尝试恢复。
```

### 第二步

```Shell

第二阶段：系统整体负载评估（2分钟）
-----------------------------
▶ 诊断命令：
   uptime
   top -bn1 | head -5

✅ 排查到该问题时的处置措施（分场景）：
   【场景A】1分钟负载 >> 5分钟负载（突增）→ 立即用 top -H -p <PID> 查看具体线程，结合 strace 追踪：
           strace -ttp <PID> -o /tmp/strace.log （观察系统调用卡在哪一步）
   【场景B】负载正常但业务慢 → 跳过系统层，直接跳至第八阶段检查应用日志。
```

### 第三步

```Shell

第三阶段：硬件与内核底层检查（硬件报错/OOM）
-----------------------------
▶ 诊断命令：
   dmesg -T | grep -E "error|fail|OOM|soft lockup|Bad page" | tail -20

✅ 排查到该问题时的处置措施：
   1. 发现 "Out of memory: Kill process" → 说明触发 OOM Killer。
      - 立即查看被杀进程：dmesg -T | grep -i "killed process"
      - 若被杀的是核心业务进程，执行：echo -1000 > /proc/<PID>/oom_score_adj （保护进程不被杀）
      - 临时增加 Swap：dd if=/dev/zero of=/swapfile bs=1G count=4 && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile
      - 彻底解决：调整 JVM/应用内存参数（-Xmx），或升级物理内存。
   2. 发现 "I/O error" 或 "Buffer I/O error" → 磁盘物理坏道。
      - 立即使用 smartctl -a /dev/sda 检查磁盘健康度（Health Status）。
      - 若故障盘为数据盘，马上 umount 并使用 dd 尝试抢救数据，同时通知机房更换硬盘。
```

### 第四步

```Shell
第四阶段：内存与 CPU 深度排查
-----------------------------
▶ 诊断命令：
   free -h
   ps aux --sort=-%mem | head -8
   mpstat -P ALL 1 5

✅ 排查到该问题时的处置措施：
   1. 物理内存用尽（Available < 5%，Swap 使用率 > 80%）：
      - 找出内存泄漏进程（Java 堆外内存泄漏常表现为 RES 持续增长）。
      - 执行 jmap -heap <PID>（Java进程）或查看 /proc/<PID>/smaps。
      - 临时止血：kill -9 <PID> 并重启该业务进程（需业务方确认）。
      - 永久修复：调整应用 Xmx 为物理内存的 60%，并加入 -XX:MaxDirectMemorySize 限制。
   2. 单核 CPU 跑满（某核心 %idle 为 0，%us 高）：
      - 执行 perf top -p <PID> 定位热点函数（C/C++）或 arthas watch（Java）。
      - 若为死循环，发送 SIGQUIT（kill -3 <PID>）打印线程栈分析阻塞点，修复代码逻辑。
   3. 内核态 CPU 高（%sy > 30%）：
      - 检查是否频繁上下文切换：vmstat 1 5 （观察 cs 列 > 10万 则异常）。
      - 调整内核参数：sysctl -w net.core.somaxconn=65535 降低锁竞争。
      - 或禁用不必要的中断合并（ethtool -C eth0 rx-usecs 0）。
```

### 第五步

```Shell



第五阶段：磁盘 I/O 与存储（最易导致系统"假死"）
-----------------------------
▶ 诊断命令：
   df -h && df -i
   iostat -x 1 5
   iotop -o (查看实际读写进程)

✅ 排查到该问题时的处置措施：
   1. 磁盘空间满（Use% = 100%）：
      - 查找超大文件：find / -type f -size +1G -exec ls -lh {} \; 2>/dev/null
      - 清理核心日志（注意：切勿直接 rm 正在被进程写入的日志文件，否则磁盘空间不释放！）
      - ✔ 正确做法：cat /dev/null > /var/log/huge.log （清空内容）或 truncate -s 0 /var/log/huge.log
      - 若 Inode 满（IUse% = 100%），说明存在海量小文件：find /tmp -type f -name "*.tmp" | xargs rm -f （清理临时缓存目录）。
   2. 磁盘 I/O 等待高（%util > 90%，await > 50ms）：
      - 修改 I/O 调度策略（SSD 建议 none/noop，机械盘建议 mq-deadline）：
        echo none > /sys/block/sdX/queue/scheduler
      - 若为数据库服务器，调大脏页回写间隔以降低磁盘冲突：
        sysctl -w vm.dirty_writeback_centisecs=6000
        sysctl -w vm.dirty_ratio=30
      - 将高读写业务（如日志存储）迁移至高速盘或内存文件系统（tmpfs）。
```

### 第六步

```Shell
第六阶段：网络连接与端口（连接耗尽/丢包）
-----------------------------
▶ 诊断命令：
   ping -c 4 网关IP
   ss -ant | awk '{print $1}' | sort | uniq -c
   iftop / sar -n DEV 1 5

✅ 排查到该问题时的处置措施：
   1. 连接堆积（TIME_WAIT 数万）：
      - 临时缓解：开启端口复用和快速回收（仅适用非 NAT 环境）：
        sysctl -w net.ipv4.tcp_tw_reuse=1
        sysctl -w net.ipv4.tcp_tw_recycle=0  # 禁用 recycle（NAT下会导致问题）
        sysctl -w net.ipv4.tcp_fin_timeout=30
      - 根本解决：修改业务代码，将短连接改为长连接（HTTP Keep-Alive / 数据库连接池）。
   2. 连接堆积（CLOSE_WAIT 暴增）：
      - 原因：应用未正确关闭 Socket，被动关闭后未调用 close()。
      - 立即重启应用服务释放连接（临时止血）。
      - 永久修复：排查代码中的 InputStream 或 Socket，确保 finally 块中执行 close()。
   3. 网卡丢包（ifconfig 显示 dropped > 0）：
      - 检查网卡缓冲区：ethtool -g eth0
      - 增大环形缓冲区：ethtool -G eth0 rx 4096 tx 4096
      - 若突发流量过高，启用 NIC 多队列（开启 RPS/RFS 软中断负载均衡）。
```

### 第七步

```Shell
第七阶段：应用进程与服务状态（进程消失/端口未监听）
-----------------------------
▶ 诊断命令：
   systemctl status <service> -l --no-pager
   lsof -p <PID> | wc -l
   cat /proc/<PID>/limits

✅ 排查到该问题时的处置措施：
   1. 进程存在但端口未监听（端口冲突）：
      - 查看谁占用了端口：ss -tlnp | grep <端口>
      - 杀死占用进程：kill -9 <PID> 并重启服务。
   2. 文件描述符泄漏（fd 数超过软限制 1024/4096）：
      - 临时扩大限制：ulimit -n 65535 （当前会话有效）。
      - 永久生效：修改 /etc/security/limits.conf，添加：
        * soft nofile 65535
        * hard nofile 65535
      - 注意：Java 进程需在启动脚本中添加 -Djava.nio.file.FileDescriptor 参数限制，或更新框架版本修复泄漏。
   3. 进程僵死（状态为 D 或 Z）：
      - D 状态（不可中断睡眠）通常卡在磁盘 I/O，无法 kill -9。只能重启宿主机或等待 I/O 超时。
      - Z 状态（僵尸进程）需 kill 其父进程（kill -HUP <PPID>）来回收。
```

### 第八步

```Shell

第八阶段：应用层日志深扒（最终真相）
-----------------------------
▶ 诊断命令（按优先级）：
   tail -500 /var/log/app/error.log | grep -B5 -A5 "FATAL\|Exception"
   tail -100 /var/log/nginx/access.log | awk '{print $9}' | sort | uniq -c （查看状态码分布）

✅ 排查到该问题时的处置措施（针对常见日志报错）：
   1. 日志报 "Connection refused" → 下游依赖（Redis/MySQL）挂了或防火墙拦截。
      - 检查下游健康状态，重启下游服务或关闭防火墙（systemctl stop firewalld）测试。
   2. 日志报 "Too many open files" → 进程句柄用尽。
      - 执行上文的 limits.conf 修改，并重启应用使配置生效。
   3. 日志报 "java.lang.OutOfMemoryError: Metaspace" → 类加载过多。
      - 增加 MetaspaceSize：-XX:MaxMetaspaceSize=512m，或排查是否存在动态生成类的代码漏洞。
   4. 日志报 "PHP Fatal error: Allowed memory size" → PHP 内存不足。
      - 修改 php.ini：memory_limit = 512M 并重载 php-fpm。
```

### 第九步

```Shell
第九阶段：终极止损三板斧（慎用，需业务审批）
-----------------------------
✅ 排查到该问题时的处置措施（按严重程度从低到高）：
   【第一板斧】热重载服务，不中断进程：
      - Nginx: nginx -s reload
      - Java: 使用 kill -HUP <PID> 触发优雅重启（需应用支持）
   【第二板斧】强制清理系统缓存（缓解内存压力）：
      - 执行前务必先 Sync 磁盘：sync && echo 3 > /proc/sys/vm/drop_caches
      - ⚠️ 注意：该操作会清空 PageCache，可能导致磁盘 I/O 瞬间飙升，建议在业务低峰期或先执行 echo 1（仅清页面缓存）。
   【第三板斧】冷重启整机：
      - 执行：reboot
      - ⚠️ 注意：需同步业务方，确保流量已切走（通过 LVS/Nginx 摘除节点）后再执行。
```

### 网络

```Shell
1. 用户反馈“访问不了”
   ↓
2. ping IP 通不通？
   ├── 不通 → 查路由（traceroute）→ 查网关/交换机/VPN
   └── 通 → 进入第3步
   ↓
3. telnet IP 端口 通不通？
   ├── 不通（超时）→ 查防火墙（firewalld/iptables/云安全组）
   ├── 不通（拒绝）→ 查服务（systemctl status）→ 查端口监听（ss -tulnp）
   └── 通 → 进入第4步
   ↓
4. 是域名访问吗？
   ├── 是 → 查 DNS（dig）→ 查 hosts 文件
   └── 否 → 查应用自身错误日志（/var/log/xxx/error.log）
   ↓
5. 如果能通但巨慢 → 查带宽（iftop）→ 查丢包（mtr）→ 查 MTU（ping -M）→ 查内核连接表
```
